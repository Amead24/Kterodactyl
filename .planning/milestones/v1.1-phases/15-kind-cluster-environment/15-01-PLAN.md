**1. hack/kind-config.yaml** -- Kind cluster configuration with port mapping:
```yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:

  image: kindest/node:v1.32.11@sha256:5fc52d52a7b9574015299724bd68f183702956aa4a2116ae75a63cb574b35af8

```
Use `listenAddress: "0.0.0.0"` (NOT `127.0.0.1`) for WSL2 compatibility. The `containerPort: 30080` MUST match `nodePort: 30080` in ci-values.yaml.

**2. hack/ci-values.yaml** -- Helm values override for the test environment:
```yaml
image:
  repository: kterodactyl
  tag: test
  pullPolicy: Never

apiService:
  type: NodePort
  port: 8080
  nodePort: 30080

manager:
  resources:
    limits:
      cpu: "1"
      memory: 256Mi
    requests:
      cpu: 100m
      memory: 128Mi
```
`pullPolicy: Never` is mandatory because kind-loaded images are not in any registry. The image name `kterodactyl:test` matches the `E2E_IMG` Makefile variable.

**3. hack/wait-for-ready.sh** -- Two-stage readiness gate:
```bash
#!/usr/bin/env bash
set -euo pipefail

NAMESPACE="${1:-kterodactyl-system}"
TIMEOUT="${2:-180s}"

echo "Waiting for deployment to be available in namespace $NAMESPACE..."
kubectl wait deployment -l app.kubernetes.io/name=kterodactyl \
  -n "$NAMESPACE" \
  --for=condition=Available \
  --timeout="$TIMEOUT"

echo "Verifying API is accessible at localhost:8080..."
for i in $(seq 1 30); do
  if curl -sf http://localhost:8080/healthz > /dev/null 2>&1; then
    echo "Kterodactyl is ready at http://localhost:8080"
    exit 0
  fi
  sleep 2
done

echo "ERROR: API did not become accessible at localhost:8080 within 60s"
echo "Debug: kubectl get pods -n $NAMESPACE"
kubectl get pods -n "$NAMESPACE"
echo "Debug: kubectl logs -l app.kubernetes.io/name=kterodactyl -n $NAMESPACE --tail=50"
kubectl logs -l app.kubernetes.io/name=kterodactyl -n "$NAMESPACE" --tail=50 2>/dev/null || true
exit 1
```
Make the script executable: `chmod +x hack/wait-for-ready.sh`. Stage 1 uses `kubectl wait` for K8s-level readiness. Stage 2 uses `curl` to confirm the NodePort chain works end-to-end. On failure, it dumps pod status and logs for debugging.

The label selector `app.kubernetes.io/name=kterodactyl` matches the Helm chart's selectorLabels from `_helpers.tpl`. The health endpoint `/healthz` is registered on the API router at port 8080 (confirmed in `internal/api/routes.go:58-59`).

**New variable:**
```makefile
E2E_IMG ?= kterodactyl:test
```

**test-e2e-setup target:**
```makefile
.PHONY: test-e2e-setup
test-e2e-setup: ## Create kind cluster with Helm-deployed Kterodactyl (accessible at localhost:8080).
	@$(KIND) delete cluster --name $(KIND_CLUSTER) 2>/dev/null || true
	$(KIND) create cluster --name $(KIND_CLUSTER) --config hack/kind-config.yaml
	$(MAKE) docker-build IMG=$(E2E_IMG)
	$(KIND) load docker-image $(E2E_IMG) --name $(KIND_CLUSTER)
	helm install kterodactyl chart/ \
		-f hack/ci-values.yaml \
		-n kterodactyl-system --create-namespace --wait --timeout 3m
	bash hack/wait-for-ready.sh kterodactyl-system 180s
	@echo "Kterodactyl is ready at http://localhost:8080"
```

Key design decisions:
- First line `kind delete cluster` ensures clean slate (idempotent -- exits 0 if cluster doesn't exist). The `2>/dev/null || true` suppresses the "cluster not found" message.
- `$(MAKE) docker-build IMG=$(E2E_IMG)` reuses the existing docker-build target with the test image name.
- `$(KIND) load docker-image` transfers the image from Docker to kind's containerd. This MUST happen before helm install.
- `helm install --wait --timeout 3m` waits for the K8s deployment to be ready (rollout complete).
- `bash hack/wait-for-ready.sh` adds the curl-based health check on top.

**test-e2e-teardown target:**
```makefile
.PHONY: test-e2e-teardown
test-e2e-teardown: ## Delete the kind cluster and all associated resources.
	$(KIND) delete cluster --name $(KIND_CLUSTER)
```

This is idempotent -- `kind delete cluster` exits 0 even if the cluster doesn't exist. Helm release and all K8s resources are destroyed with the cluster.

Do NOT remove or modify the existing `setup-test-e2e`, `test-e2e`, or `cleanup-test-e2e` targets. The new targets coexist alongside them. Future phases may update or consolidate them.
