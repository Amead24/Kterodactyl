**Setup function `setupTestServer(t *testing.T) (*httptest.Server, *auth.InviteService)`:**
- Create JWT service with test signing key (`[]byte("test-signing-key-32-bytes-long!!")`, 24h expiry)
- Create fake K8s client with `corev1` and `gamev1alpha1` schemes, `WithStatusSubresource(&gamev1alpha1.GameServer{})`
- Create `auth.InviteService` with fake client, namespace `"kterodactyl-system"`, nil SMTP, panel URL `"https://panel.test"`
- Create `auth.UserStore` with fake client and namespace `"kterodactyl-system"`
- Create manifest loader using a helper `createTestManifestLoader(t)` that creates a temp dir with `minecraft/manifest.yaml` (copy the manifest content from research Example 4 -- needs name, displayName, image, ports, parameters with EULA+TYPE, resources, and parameterSchema)
- Construct `api.NewServer(api.Config{...})` with all dependencies, `OperatorNamespace: "kterodactyl-system"`, `BindAddress: ":0"`
- Create `httptest.NewServer(srv.HTTPServer().Handler)`
- Register `t.Cleanup(ts.Close)`
- Return the test server and invite service

**Helper functions (local to the test file):**
- `jsonPost(t, client, url string, body interface{}) *http.Response` -- marshal body to JSON, POST with Content-Type application/json
- `jsonPostAuth(t, client, url string, body interface{}, token string) *http.Response` -- same as jsonPost but adds Authorization: Bearer header
- `jsonGetAuth(t, client, url, token string) *http.Response` -- GET with Authorization header
- `jsonDeleteAuth(t, client, url, token string) *http.Response` -- DELETE with Authorization header
- `assertStatus(t, resp *http.Response, expected int)` -- compare status code, on failure read and print body for diagnostics
- `decodeJSONResponse(t, resp *http.Response) map[string]interface{}` -- decode response body into map (name this differently from the `decodeJSON` in `internal/api/request.go` since this is a different package anyway, but avoid confusion)

**`TestAPILifecycle(t *testing.T)` function:**
1. Call `setupTestServer(t)` to get `ts` and `inviteSvc`
2. Use `ts.Client()` as the HTTP client (follows redirects, handles cookies if needed)
3. **Pre-seed:** Create invite via `inviteSvc.CreateInvite(ctx, "alice@test.com", "bootstrap", 72)` -- this returns an `*auth.Invite` struct with `.Token` field
4. **Step 1 - Register:** POST to `ts.URL + "/api/v1/auth/register"` with body `{"username": "alice", "email": "alice@test.com", "password": "securepassword123", "inviteToken": invite.Token}`. Assert 201. Decode response, extract `"token"` field as JWT string. Fatal if token is empty.
5. **Step 2 - Create server:** POST to `ts.URL + "/api/v1/gameservers"` with body `{"name": "my-mc-server", "gameType": "minecraft", "parameters": {"EULA": "TRUE", "TYPE": "VANILLA"}}`, auth with token from step 1. Assert 201. Decode response, verify `"name"` is `"my-mc-server"` and `"gameType"` is `"minecraft"`.
6. **Step 3 - Get server:** GET `ts.URL + "/api/v1/gameservers/my-mc-server"` with auth. Assert 200. Decode response, verify `"gameType"` is `"minecraft"` and `"name"` is `"my-mc-server"`.
7. **Step 4 - Delete server:** DELETE `ts.URL + "/api/v1/gameservers/my-mc-server"` with auth. Assert 204.
8. **Step 5 - Verify deleted:** GET `ts.URL + "/api/v1/gameservers/my-mc-server"` with auth. Assert 404.

**Important notes:**
- Use `interface{}` (or `any`) for the body parameter in helper functions so we can pass both `map[string]string` and `map[string]interface{}` (the create server request has nested parameters)
- The `createTestManifestLoader` helper must include the full `parameterSchema` section with EULA (const "TRUE") and TYPE (enum) because `handleCreateGameServer` validates parameters against the schema
- Do NOT import internal response types -- use `map[string]interface{}` for true blackbox testing
- Each step depends on the previous one's output, so this is one linear test function (not subtests)

**Key details:**
- `-tags integration` matches the `//go:build integration` build tag in the test file
- `-v` for verbose output (shows test names and pass/fail)
- `-count=1` disables test caching (integration tests should always run fresh)
- No build prerequisites needed (no `manifests`, `generate`, etc.) -- integration tests use fake K8s clients, not envtest
- The existing `make test` uses `go list ./... | grep -v /e2e` which does NOT filter `/integration`, but the `//go:build integration` build tag prevents it from being picked up by `go test` without `-tags integration`. This is the same isolation pattern used by e2e tests.
