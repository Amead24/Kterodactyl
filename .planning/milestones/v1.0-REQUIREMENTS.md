# Requirements: Kterodactyl

**Defined:** 2026-02-09
**Core Value:** Admins can deploy a single Helm chart and give their users self-service game server provisioning backed entirely by Kubernetes

## v1 Requirements (All Complete)

### Operator & Infrastructure (7/7)

- [x] **OPER-01**: Operator creates and manages GameServer CRD with v1alpha1 API
- [x] **OPER-02**: GameServer follows state machine lifecycle (Creating -> Ready -> Allocated -> Shutdown)
- [x] **OPER-03**: User can start, stop, restart, and delete game servers via API
- [x] **OPER-04**: Admin can set global resource limits (max servers, CPU/RAM per server)
- [x] **OPER-05**: Each user's servers run in isolated namespace with ResourceQuotas and NetworkPolicies
- [x] **OPER-06**: GameServer CRDs are GitOps-compatible (manageable via `kubectl apply`)
- [x] **OPER-07**: Operator deploys as a single binary with leader election for HA

### Networking & DNS (4/4)

- [x] **NET-01**: Each game server is accessible at `<game>.<username>.domain.com`
- [x] **NET-02**: DNS Controller creates Gateway API HTTPRoute resources for wildcard routing
- [x] **NET-03**: ExternalDNS integration automatically provisions DNS records
- [x] **NET-04**: User sees connection info (DNS name + port) in UI after server is ready

### Authentication (4/4)

- [x] **AUTH-01**: Admin can invite users via email
- [x] **AUTH-02**: User can self-register with email and password
- [x] **AUTH-03**: User sessions persist via JWT tokens across browser refresh
- [x] **AUTH-04**: User can only access and manage their own game servers

### Game Definitions (5/5)

- [x] **GAME-01**: Games are defined declaratively (Dockerfile + manifest.yaml per game in games/ directory)
- [x] **GAME-02**: Game manifest defines configurable parameters with JSON schema (ports, env vars, settings)
- [x] **GAME-03**: Minecraft Java Edition ships as reference game definition
- [x] **GAME-04**: UI dynamically generates configuration forms from game manifest schemas
- [x] **GAME-05**: Documentation covers how to contribute new game definitions via PR

### Console & Monitoring (3/3)

- [x] **CONS-01**: User can view real-time server console output via WebSocket
- [x] **CONS-02**: User can send commands to running game server via console
- [x] **CONS-03**: User can see CPU, RAM, and disk usage for their servers in real-time

### Mod Support (3/3)

- [x] **MOD-01**: User can upload mod files to a game server via the UI
- [x] **MOD-02**: Mods are stored on a separate PersistentVolume mounted to the game server container
- [x] **MOD-03**: Server automatically restarts after mod upload completes

### Backups (5/5)

- [x] **BKUP-01**: User can trigger an on-demand backup of their game server
- [x] **BKUP-02**: Admin can configure scheduled backups via cron schedule
- [x] **BKUP-03**: Backups are stored in S3-compatible storage (MinIO, AWS S3, GCS)
- [x] **BKUP-04**: Backup CRD tracks backup status, size, and S3 location
- [x] **BKUP-05**: Admin can restore a game server from a backup

### Observability (4/4)

- [x] **OBS-01**: Operator exposes Prometheus metrics (game server count by state, reconciliation latency)
- [x] **OBS-02**: API server exposes Prometheus metrics (request rate, latency, error rate)
- [x] **OBS-03**: ServiceMonitor CRDs are created for Prometheus Operator autodiscovery
- [x] **OBS-04**: Metrics use low-cardinality labels only (game_type, server_state -- not user_id or pod_name)

### Helm Chart (4/4)

- [x] **HELM-01**: Kterodactyl installs via a single `helm install` command
- [x] **HELM-02**: Helm values are configurable for Gateway API vs Ingress, storage class, and domain
- [x] **HELM-03**: CRDs install via Helm crds/ directory with proper ordering
- [x] **HELM-04**: Chart supports both homelab (single node) and multi-node cluster deployments

### Documentation (4/4)

- [x] **DOCS-01**: Docusaurus site covers installation, configuration, and usage
- [x] **DOCS-02**: Game definition contribution guide with Minecraft example walkthrough
- [x] **DOCS-03**: Helm values reference with all configurable options documented
- [x] **DOCS-04**: Architecture overview for contributors

## v2 Requirements (Deferred)

### Authentication

- **AUTH-05**: OIDC/SSO integration (Google, Steam, Apple via Dex/Keycloak)
- **AUTH-06**: Subuser RBAC (share server access with friends, granular permissions)

### File Management

- **FILE-01**: Web-based file manager (edit configs in browser, upload/download)
- **FILE-02**: SFTP access for power users

### User Features

- **USER-01**: User can view server logs and download log files
- **USER-02**: User can download their own backups
- **USER-03**: Scheduled tasks (automate restarts, commands on schedule)

### Scale

- **SCALE-01**: Multi-tenancy with organizations
- **SCALE-02**: Per-user resource quotas (individual budgets)
