**Important:** The EnsureSigningKey and LoadAdminConfig calls use context.Background() because they run before the manager starts. The manager hasn't started its caches yet at this point, but these functions use direct API calls (Get, not cached). Actually, wait -- mgr.GetClient() returns a client that uses the cache, which isn't started yet. We need to use a direct client for pre-start operations.

**Correction:** Before the manager starts, the cache isn't running. So for EnsureSigningKey and LoadAdminConfig, we need to create a direct client. However, looking at the existing code, the controllers don't have this issue because they run after manager starts.

**Alternative approach (simpler):** Move the auth initialization into a Runnable that runs after manager start. But this adds complexity.

**Simplest approach:** Use the manager's GetAPIReader() which returns a direct reader (not cached). But we need both read and write for EnsureSigningKey.

**Actually:** Looking at controller-runtime's manager.Server, the server's Start() is called when the manager starts. So we can defer the auth initialization to the Server's construction time by not calling EnsureSigningKey until the manager is running. But NewServer is called before Start().

**Pragmatic solution:** Create a direct K8s client for the bootstrap operations:
```go
directClient, err := client.New(ctrl.GetConfigOrDie(), client.Options{Scheme: scheme})
if err != nil {
    setupLog.Error(err, "failed to create direct client for bootstrap")
    os.Exit(1)
}
```
Use `directClient` for EnsureSigningKey and LoadAdminConfig (pre-start operations). Use `mgr.GetClient()` for the API Server config (runtime operations that happen after cache is started).

Add import: "sigs.k8s.io/controller-runtime/pkg/client"

This is the correct pattern: bootstrap with direct client, runtime with cached client.
