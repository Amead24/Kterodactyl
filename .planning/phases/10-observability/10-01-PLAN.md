<task type="auto">
  <name>Task 1: Create centralized metrics package with all Prometheus metric definitions</name>
  <files>internal/metrics/metrics.go</files>
  <action>
Create `internal/metrics/metrics.go` with ALL metric definitions for the entire phase (both operator and API server metrics). Register all metrics with controller-runtime's `metrics.Registry` in an `init()` function.

Define these metrics (all prefixed with `kterodactyl_`):

1. `kterodactyl_gameservers_by_state` — GaugeVec with labels `["state", "game_type"]`. Tracks current count of GameServers by state and game type.
2. `kterodactyl_reconciliation_duration_seconds` — HistogramVec with labels `["controller"]` and buckets `{0.01, 0.05, 0.1, 0.25, 0.5, 1.0, 2.5, 5.0, 10.0}`. Tracks reconciliation duration.
3. `kterodactyl_http_requests_total` — CounterVec with labels `["method", "route", "status_code"]`. Tracks total HTTP requests.
4. `kterodactyl_http_request_duration_seconds` — HistogramVec with labels `["method", "route", "status_code"]` and `prometheus.DefBuckets`. Tracks HTTP request latency.
5. `kterodactyl_http_requests_inflight` — Gauge (no labels). Tracks concurrent in-flight HTTP requests.

Import `crmetrics "sigs.k8s.io/controller-runtime/pkg/metrics"` for registry access. Do NOT use `prometheus.MustRegister()` (that uses the default registry, which is wrong).

All labels must be low-cardinality per OBS-04: `state` (6 enum values), `game_type` (bounded by game definitions), `controller` (2 values: "gameserver", "backup"), `method` (GET/POST/PUT/DELETE), `route` (chi route patterns), `status_code` (HTTP status codes).
  </action>
  <verify>Run `go build ./internal/metrics/` to confirm the package compiles without errors.</verify>
  <done>
`internal/metrics/metrics.go` exists with 5 metric definitions, all registered via `crmetrics.Registry.MustRegister()` in `init()`. No use of the default prometheus registry.
  </done>

<task type="auto">
  <name>Task 2: Instrument GameServer reconciler with gauge and histogram metrics</name>
  <files>internal/controller/gameserver_controller.go</files>
  <action>
Add two metric recording points to the GameServerReconciler:

1. **Reconciliation duration** — At the top of the `Reconcile` method (line ~328), add a defer that records elapsed time:
   ```go
   start := time.Now()
   defer func() {
       metrics.ReconciliationDuration.WithLabelValues("gameserver").Observe(time.Since(start).Seconds())
   }()
   ```
   Place this AFTER the logger setup but BEFORE any early returns (except the not-found check — duration of a no-op is still useful).

2. **Game server count gauge** — Add a new method `updateGameServerGauge(ctx context.Context)` that:
   - Lists ALL GameServer resources across all namespaces (`r.List(ctx, gsList)`)
   - Calls `metrics.GameServersByState.Reset()` to clear stale label combinations
   - Counts servers by `(state, gameType)` pair
   - Sets gauge values with `metrics.GameServersByState.WithLabelValues(state, gameType).Set(count)`
   - Logs errors but does NOT return them (metrics must never fail reconciliation)

   Call `r.updateGameServerGauge(ctx)` at the END of the Reconcile method, just before the final return (after the state machine switch). This ensures the gauge reflects the latest state after each reconciliation.

Add import for `"github.com/kterodactyl/kterodactyl/internal/metrics"`. The `time` package is already imported.

Important: Do NOT let metric errors propagate — wrap in defensive code. A metrics failure must never prevent a GameServer from reconciling.
  </action>
  <verify>Run `go build ./internal/controller/` and `go build ./cmd/` to confirm the controller and main binary compile. Run `go vet ./internal/...` for correctness.</verify>
  <done>
`Reconcile` method records reconciliation duration via defer. `updateGameServerGauge` method exists and is called at end of Reconcile. Both use `internal/metrics` package. Build succeeds.
