    **2. Write gameserver_controller_test.go with these test cases:**

    Use Ginkgo Describe/Context/It blocks with Gomega assertions.

    **Test Case 1: "should create a Pod when GameServer is created"**
    - Create a namespace "test-ns-1" for the test
    - Create a GameServer CR with gameType "minecraft", image "itzg/minecraft-server:latest", and label `kterodactyl.io/owner: testuser1`
    - Use Eventually (10s timeout, 250ms poll) to assert:
      - Pod with same name exists in same namespace
      - Pod has container named "gameserver" with correct image
      - Pod has owner reference pointing to the GameServer
      - Pod has labels matching util.GameServerLabels("testuser1", "minecraft")
    - Clean up: delete GameServer CR

    **Test Case 2: "should transition through Creating to Starting state"**
    - Create a GameServer CR
    - Use Eventually to assert Status.State transitions from "" to Creating to Starting
    - Note: In envtest, Pods never actually become Running (no kubelet), so we cannot test the Starting -> Ready transition in unit tests. Document this limitation.
    - Clean up: delete GameServer CR

    **Test Case 3: "should handle deletion with finalizer cleanup"**
    - Create a GameServer CR
    - Wait for finalizer to be added (Eventually check for finalizer in metadata.finalizers)
    - Wait for Pod to be created
    - Delete the GameServer CR
    - Use Eventually to assert:
      - GameServer CR is gone (Get returns NotFound)
      - Pod is gone (Get returns NotFound)

    **Test Case 4: "should set Error state when owner label is missing"**
    - Create a GameServer CR WITHOUT the `kterodactyl.io/owner` label
    - Use Eventually to assert Status.State becomes Error
    - Clean up: delete GameServer CR

    **Test Case 5: "should create user namespace with ResourceQuota"**
    - Create a GameServer CR with owner label "quotauser"
    - Use Eventually to assert:
      - Namespace "user-quotauser" exists
      - Namespace has label `kterodactyl.io/managed-by: kterodactyl`
      - ResourceQuota "user-quota" exists in namespace "user-quotauser"
      - ResourceQuota has hard limits set (pods field present)
    - Clean up: delete GameServer CR and namespace

    **Test Case 6: "should create NetworkPolicy in user namespace"**
    - Create a GameServer CR with owner label "netpoluser"
    - Use Eventually to assert:
      - NetworkPolicy "deny-cross-namespace" exists in "user-netpoluser" namespace
      - NetworkPolicy has both Ingress and Egress policy types
    - Clean up: delete GameServer CR and namespace

    **Test Case 7: "should use admin ConfigMap values for quota when available"**
    - Create ConfigMap "kterodactyl-admin-config" in "test-system" namespace with custom values (maxServersPerUser: "2", quotaCPURequests: "2")
    - Create a GameServer CR with owner label "configuser"
    - Use Eventually to assert:
      - ResourceQuota in "user-configuser" has the custom CPU request value
    - Clean up: delete ConfigMap, GameServer CR, namespace

    **General test patterns:**
    - Use unique namespaces per test to avoid interference
    - Use AfterEach to clean up created resources
    - Use Eventually with 10s timeout for all async assertions
    - Use Consistently (2s) where appropriate to verify things do NOT happen

    **3. Run `make test` and fix any failures.**

    - Pods never start (no kubelet/scheduler), so Starting -> Ready transition cannot be tested
    - This is expected and documented; a kind cluster test in CI will cover the full lifecycle
    - Tests focus on: CR -> Pod creation, state machine transitions that don't require running Pods, finalizer cleanup, namespace isolation resources
