    Write 8 markdown documentation files. All content must use code blocks for any text containing angle brackets or curly braces (MDX strictness). Use Docusaurus admonitions (:::tip, :::warning, :::info) where appropriate.

    **Getting Started (3 files):**

    1. **overview.md**: Front matter with `sidebar_position: 1`. Explain what Kterodactyl is: a Kubernetes-native game server management panel, open-source alternative to Pterodactyl. Explain how it differs (replaces Wings+Docker+Postgres with CRDs, operator, K8s primitives). Describe the two target user personas (homelab admins, service operators). List key features: CRD-based lifecycle, community game definitions, dynamic UI from JSON Schema manifests, Gateway API routing, JWT auth, mod support, S3 backups, Prometheus metrics, Helm chart installation. Include TODO-01 content (how Kterodactyl differs from Agones -- Agones is for multiplayer fleet scaling, Kterodactyl is for self-service provisioning of dedicated servers).

    2. **prerequisites.md**: Front matter with `sidebar_position: 2`. List: Kubernetes 1.26+ cluster, Helm 3.x, kubectl configured, Gateway API CRDs installed (link to official docs), a wildcard DNS entry pointing to cluster ingress (explain briefly), optionally S3-compatible storage for backups (MinIO for homelab), optionally SMTP server for email invitations.

    3. **installation.md**: Front matter with `sidebar_position: 3`. Walkthrough: `helm install kterodactyl ./chart` (or OCI registry when published). Show minimal install, then show install with custom values (baseDomain, backup enabled). Cover post-install steps derived from NOTES.txt: accessing the API (port-forward or LoadBalancer), JWT key auto-generation, creating secrets for S3 and SMTP if enabled, CRD upgrade note, bootstrapping first admin user. Use admonition for CRD upgrade warning.

    **Configuration (5 files):**

    4. **helm-values.md**: Front matter with `sidebar_position: 1`. Complete reference table for ALL values in chart/values.yaml. Organize into sections matching values.yaml structure: Core Settings (replicaCount, image, imagePullSecrets, nameOverride, fullnameOverride), ServiceAccount, Manager, Scheduling (nodeSelector, tolerations, affinity), API Service, Metrics, ServiceMonitor, NetworkPolicy, AdminConfig (limits, quota, containerDefaults, networking, auth, smtp, storage, backup). Each section has a markdown table with Parameter, Description, Default columns. Derive descriptions from the `# --` comments in values.yaml.

    5. **admin-config.md**: Front matter with `sidebar_position: 2`. Explain the kterodactyl-admin-config ConfigMap. Describe that the operator loads it per-reconciliation (no restart needed). Document each section: limits (maxServersGlobal, maxServersPerUser), quota (cpuRequests, cpuLimits, memoryRequests, memoryLimits, pods, pvcs, storage), containerDefaults (cpu, memory, requestCPU, requestMemory, maxCPU, maxMemory, minCPU, minMemory), networking (baseDomain, gateway name/namespace/controllerNamespace), auth (jwtExpirationHours, inviteExpirationHours, registrationEnabled, panelURL), smtp (host, port, username, from), storage (modStorageClass, modStorageSize), backup (s3 endpoint/bucket/region/useSSL, retentionCount). Explain that the Helm chart generates this ConfigMap from adminConfig values.

    6. **networking.md**: Front matter with `sidebar_position: 3`. Explain DNS pattern: `game.username.baseDomain`. Describe Gateway API requirement (HTTPRoute resources created by DNS controller). Explain baseDomain configuration (empty = DNS disabled). Describe gateway settings (name, namespace, controllerNamespace). Note that Cloudflare Tunnel or ExternalDNS can provide wildcard DNS. Mention NetworkPolicy creation for game server pods.

    7. **backups.md**: Front matter with `sidebar_position: 4`. Explain S3-compatible backup system. Describe setup: enable in values.yaml, create S3 credentials secret (kubectl command), configure endpoint/bucket/region. Explain backup flow: operator tars game data from pod, gzips, uploads to S3. Describe retention (retentionCount). Note MinIO for homelab, AWS S3/GCS for cloud. Note that bucket is auto-created on first backup.

    8. **auth.md**: Front matter with `sidebar_position: 5`. Explain JWT authentication (HS256, signing key auto-generated and stored in K8s Secret). Describe user model (stored in K8s Secrets). Explain admin invite flow (admin creates invite, user registers with invite token). Describe configurable settings (JWT expiration, invite expiration, registration toggle). Explain SMTP setup for email invitations (optional -- invite link returned in API response even without SMTP).

    After writing all files, verify build: `cd docs-site && npm run build`
