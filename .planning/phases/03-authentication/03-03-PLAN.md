  <name>Task 1: Create auth middleware and invite service</name>
  <files>internal/auth/middleware.go, internal/auth/invite.go</files>
  <action>
  Add the go-mail dependency:
  ```
  go get github.com/wneessen/go-mail
  ```

  **Create `internal/auth/middleware.go`:**
  - Package `auth`
  - Define `contextKey` type (unexported `type contextKey string`)
  - Define `ContextKeyUser contextKey = "user"` (exported)
  - Define `AuthMiddleware` struct with `jwtService *JWTService`
  - `NewAuthMiddleware(jwtService *JWTService) *AuthMiddleware`

  Methods:
  - `Authenticate(next http.Handler) http.Handler`:
    - Extract Bearer token from Authorization header using `extractBearerToken(r)`
    - If empty: return 401 "missing authorization header"
    - Validate token via `jwtService.ValidateToken()`
    - If invalid: return 401 "invalid or expired token"
    - Set claims in context: `context.WithValue(r.Context(), ContextKeyUser, claims)`
    - If `jwtService.ShouldRefresh(claims)`: generate new token, set `X-Refresh-Token` response header
    - Call `next.ServeHTTP(w, r.WithContext(ctx))`
  - `RequireAdmin(next http.Handler) http.Handler`:
    - Extract claims from context via `GetUserFromContext()`
    - If claims nil or Role != RoleAdmin: return 403 "admin access required"
    - Call next

  Helper functions:
  - `extractBearerToken(r *http.Request) string` -- strips "Bearer " prefix from Authorization header
  - `GetUserFromContext(ctx context.Context) *KterodactylClaims` -- type-asserts ContextKeyUser from context

  All error responses should use `http.Error()` with JSON-like message format for consistency with Phase 4 API.

  **Create `internal/auth/invite.go`:**
  - Package `auth`
  - Import `github.com/wneessen/go-mail` (aliased as `mail`)

  Define `SMTPConfig` struct:
  - Host, From, Username, Password string; Port int

  Define `InviteService` struct:
  - `client client.Client`
  - `namespace string`
  - `smtp *SMTPConfig` (nil if SMTP not configured)
  - `panelURL string`

  Constructor:
  - `NewInviteService(client client.Client, namespace string, smtp *SMTPConfig, panelURL string) *InviteService`
  - smtp can be nil (email disabled)

  Define `Invitation` struct:
  - Token, Email, InvitedBy, ExpiresAt string

  Methods:
  - `CreateInvite(ctx context.Context, email, invitedBy string, expirationHours int) (*Invitation, error)`:
    - Generate 32-byte random token via `crypto/rand`, hex-encode to 64-char string
    - Compute expiration time as `time.Now().Add(time.Duration(expirationHours) * time.Hour)`
    - Create K8s Secret named `invite-<first-12-chars-of-token>` in operator namespace
    - Labels: `kterodactyl.io/managed-by: kterodactyl`, `kterodactyl.io/resource-type: invite`
    - Data keys: `token` (full 64-char hex), `email`, `invited-by`
    - Annotation: `kterodactyl.io/expires-at: <RFC3339>`
    - If SMTP configured: send email with registration link `<panelURL>/register?token=<token>`
    - If SMTP not configured: log info "SMTP not configured, returning invite link in response"
    - Return Invitation struct (including the full token)

  - `RedeemInvite(ctx context.Context, token string) (string, error)`:
    - List invite Secrets matching label `resource-type: invite`
    - Find the Secret where `Data["token"]` matches the provided token
    - If not found: return `ErrInvalidToken`
    - Check expiration annotation: if past `time.Now()`, delete the Secret and return `ErrInviteExpired`
    - Delete the Secret immediately (single-use: validate-then-delete)
    - Return the email from the Secret data

  Helper:
  - `sendInviteEmail(email, token string) error` -- uses go-mail to send via SMTP. Create `mail.NewMsg()`, set From, To, Subject "You've been invited to Kterodactyl", HTML body with registration link. Use `mail.NewClient()` with TLS.

  The invite flow pattern:
  1. Admin calls CreateInvite -> token stored as Secret, email sent (or link returned)
  2. User provides token + username + password to registration endpoint
  3. Registration endpoint calls RedeemInvite -> validates and deletes token, returns email
  4. Registration endpoint creates user via UserStore with the redeemed email
  5. Registration endpoint generates JWT via JWTService
  (Steps 2-5 are Phase 4 API server responsibility. This plan only provides the building blocks.)

  <done>

  - Auto-refresh: new token issued in X-Refresh-Token header when nearing expiry
  - RequireAdmin blocks non-admin users with 403

  - InviteService sends email via SMTP if configured, otherwise returns link
  - RedeemInvite validates, checks expiry, and atomically deletes the invite Secret

<task type="auto">
  <name>Task 2: Create comprehensive auth package unit tests</name>
  <files>internal/auth/auth_test.go</files>
  <action>
  Create `internal/auth/auth_test.go` with tests covering the entire auth package.

  Use standard Go testing (not Ginkgo -- the auth package is a standalone library, not a controller integration).

  **Password tests:**
  - `TestHashPassword_ProducesPHCFormat`: hash a password, verify output matches `$argon2id$v=19$m=65536,t=1,p=4$...$...` pattern
  - `TestHashPassword_UniqueSalts`: hash same password twice, verify hashes differ (unique salts)
  - `TestVerifyPassword_CorrectPassword`: hash then verify with same password, expect true
  - `TestVerifyPassword_WrongPassword`: hash then verify with different password, expect false
  - `TestVerifyPassword_InvalidFormat`: pass garbage string, expect error

  **Username validation tests:**
  - `TestValidateUsername_ValidNames`: table-driven with valid names ("alice", "bob-123", "a", "a-b-c")
  - `TestValidateUsername_InvalidNames`: table-driven with invalid names ("", "UPPERCASE", "-starts-with-dash", "ends-with-dash-", "has spaces", "has.dots", strings longer than 63 chars)
  - `TestValidateUsername_ReservedNames`: table-driven with reserved names ("admin", "system", "operator", "default", "kube")

  **JWT tests:**
  - `TestJWTService_GenerateAndValidate`: create service with known key, generate token for user, validate it, verify claims match
  - `TestJWTService_ExpiredToken`: create service with 0-duration expiration, generate token, wait 1ms, validate should fail
  - `TestJWTService_InvalidSigningKey`: generate with one key, validate with different key, expect error
  - `TestJWTService_ShouldRefresh`: create token expiring in 1 hour (within 2hr threshold), verify ShouldRefresh returns true; create token expiring in 12 hours, verify returns false
  - `TestJWTService_NamespaceClaim`: verify the Namespace claim equals `util.UserNamespace(username)`

  **Middleware tests:**
  - Use `httptest.NewRecorder()` and `httptest.NewRequest()` for HTTP testing
  - `TestAuthMiddleware_ValidToken`: set Authorization header with valid Bearer token, verify next handler called, verify claims in context
  - `TestAuthMiddleware_MissingHeader`: no Authorization header, verify 401 response
  - `TestAuthMiddleware_InvalidToken`: set garbage token, verify 401 response
  - `TestAuthMiddleware_RefreshHeader`: create token near expiry, verify X-Refresh-Token header set in response
  - `TestRequireAdmin_AdminUser`: set admin claims in context, verify next handler called
  - `TestRequireAdmin_NonAdminUser`: set user claims in context, verify 403 response

  Do NOT test InviteService or UserStore with real K8s client here -- those require envtest which is expensive. They will be integration-tested in Phase 4 when the API server exercises the full flow. The unit tests here verify the logic that can be tested without a K8s cluster (password, JWT, middleware, validation).

  For middleware tests, create a real JWTService with a test signing key and generate real tokens to test with.
