<task type="auto">
  <name>Task 2: Write server CRUD E2E tests (create, verify in list, delete)</name>
  <files>e2e/tests/servers.spec.ts</files>
  <action>
Create `e2e/tests/servers.spec.ts` that imports `{ test, expect }` from `../fixtures/auth`.

Use `test.describe.serial('Game Servers', ...)` to ensure tests run in order since they share state (server created in one test is deleted in another).

Generate a unique server name at the describe level: `const serverName = 'e2e-test-' + Date.now().toString(36)` (alphanumeric, starts with letter, valid DNS label).

**Test 1: "user can create a game server" (PW-04)**
- Uses `userPage` fixture
- Navigate to `/servers/create`
- Assert: heading "Create Server" is visible (create-server.tsx line 51)
- The first step is game selection. Click the Minecraft card: `page.getByRole('link', { name: /Minecraft/i })` or `page.getByText('Minecraft Java Edition')` (from GameSelectCard which shows `game.displayName`)
- Wait for the config form to appear (URL changes to `/servers/create?game=minecraft`)
- Fill server name: `page.getByLabel('Server Name')` with the unique `serverName`
- Click "Create Server" button: `page.getByRole('button', { name: 'Create Server' })`
- Wait for navigation to server detail page: `expect(page).toHaveURL(new RegExp('/servers/' + serverName))`
- OR verify success toast: `page.getByText('created successfully')` is visible

**Test 2: "server appears in the server list" (PW-04)**
- Uses `userPage` fixture
- Navigate to `/servers`
- Assert: `page.getByText(serverName)` is visible (the server card title shows the name)
- Assert: heading "My Servers" is visible

**Test 3: "user can delete a game server" (PW-05)**
- Uses `userPage` fixture
- Navigate to `/servers`
- Find the server card containing the server name: locate the card with text matching `serverName`
- Click the delete button (Trash2 icon button). The delete button is in the CardFooter of the server card. Strategy: find the card by its title link text, then find the destructive-colored button within the closest card ancestor. Use: `page.locator('[class*="card"]').filter({ hasText: serverName }).getByRole('button').last()` — the last button in the card footer is the trash button (it has `text-destructive` class and contains the Trash2 icon, no text label).
  Alternative: `page.locator('article, [class*="card"]').filter({ hasText: serverName }).locator('button.text-destructive')` or match by the destructive class pattern.
  Simplest reliable approach: find the text, go up to the card, find all buttons, click the one with destructive styling.
- Assert: `page.getByText(serverName)` is NOT visible (with reasonable timeout for React Query 5s polling to refresh): `await expect(page.getByText(serverName)).not.toBeVisible({ timeout: 15_000 })`

**Important implementation details:**
- Server list uses React Query with 5s refetchInterval (use-servers.ts), so after deletion the list auto-refreshes within 5s
- Delete button has no confirmation dialog (server-card.tsx line 92: directly calls `deleteMutation.mutate`)
- The server card structure (server-card.tsx): Card > CardHeader (title link + status badge) > CardContent > CardFooter (start/stop buttons + trash button). The trash button is always rendered (line 87-98), has `variant="ghost"` and `className="ml-auto text-destructive hover:text-destructive"` and contains only a Trash2 icon (no text)
- The game selection step shows cards with `game.displayName` as the title (e.g., "Minecraft Java Edition")
- After selecting a game, URL becomes `/servers/create?game=minecraft` and shows the config form with "Server Name" label and "Create Server" button
- The GameConfigForm component (game-config-form.tsx) always has a "Create Server" button whether or not there's a parameter schema
